# Security Policy

## Environment-file safety notice

Grove releases `v0.1.0` through `v0.5.1` contain a destructive environment-file synchronization bug. When Grove synchronized a worktree whose path was also the canonical project root—most notably while opening, entering, or recreating the main worktree—it could unlink a configured canonical environment file and replace it with a self-referential symlink. The symlink does not contain the deleted file's contents.

Use the recommended urgent `v0.5.2` patch release or newer. Until that patch is installed, do not use an affected Grove version to create, open, or enter the main/default-branch worktree when `env_files` or service `env_file` entries are configured.

If an affected command has already run:

1. Stop running Grove against that checkout.
2. Inspect configured environment paths for self-referential, cyclic, broken, or dangling symlinks without following them.
3. Restore lost canonical contents from a backup, password manager, deployment system, or another trusted source. Grove cannot recover unlinked contents from the replacement symlink.
4. Do not replace a canonical source from an unverified linked-worktree file.

## Filesystem safety guarantees after the patch

- Grove never modifies a canonical environment source.
- When the project root and worktree are the same location, Grove does not create, replace, write, symlink, or delete any configured env file or generated local. Existing and missing locals remain untouched; in-memory port values may still feed tmux/shell environments.
- Linked worktrees get validated symlinks and marker-owned local files written atomically.
- A genuinely missing configured source remains optional; Grove skips it and leaves any destination untouched.
- Broken, cyclic, dangling, unreadable, or otherwise invalid sources abort workspace setup.
- Grove never overwrites a user-managed regular destination where a symlink is expected.
- Incorrect worktree symlinks are staged and replaced atomically only after the canonical source is validated.
- Managed local files begin with `# Generated by grove — do not edit`; only files with that marker are atomically regenerated, with values quoted/escaped.
- Existing symlinks, unusual filesystem objects, and unmarked regular files at managed-local paths are preserved and rejected.
- Global/service environment-key collisions and unknown YAML fields are rejected.
- Durable active-worktree port assignments are stored under the Git common state directory at `grove/ports.json`, guarded by a cross-process mutation lock; malformed state or config drift fails closed, and the registry contains no secret values.
- Hook scripts must resolve inside the project root, receive only a minimal env allowlist plus `hooks.env_passthrough` and Grove variables, default to a 2m timeout with 1h max, and terminate their process tree on cancellation. Child output is suppressed by default and never included in errors; `hooks.output: stream` is an explicit potentially secret-bearing opt-in. `pre_delete` timeout/failure blocks mutation; `post_create` failure remains a warning.
- Grove manages only labeled tmux targets; legacy unlabeled-name fallback is removed. New target setup is transactional, and deleting a last window preserves the containing session with a placeholder or fails closed.
- `delete --dry-run` previews mutations without running remote/unpushed checks or creating state, clean/delete partial failures return nonzero, and JSON errors use `{error:{code,message}}`. Grove refuses direct recursive deletion when Git no longer recognizes a stale worktree directory. Create rollback is limited to failures before hooks or tmux can produce worktree side effects; tmux setup failures retain the worktree and branch. Delete and clean revalidate branch tips before removal. `clean --force` skips confirmation but preserves dirty worktrees; only the explicit `clean --discard-changes` flag permits forced dirty-worktree removal.
- Environment values are not included in Grove diagnostics or errors.

## Reporting a vulnerability

Please report suspected vulnerabilities privately to the repository owner before opening a public issue. Include the Grove version, command, operating system, and a minimal reproduction that uses synthetic files. Never include environment-file contents, credentials, tokens, or other secrets.
